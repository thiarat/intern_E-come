# 1. ภาพรวมโปรเจกต์

## 1.1 วัตถุประสงค์

พัฒนาเว็บแอปพลิเคชันสำหรับขายสินค้าออนไลน์พร้อมระบบบริหารจัดการการจัดส่ง โดยกำหนดให้ใช้เทคโนโลยีและ API ที่ไม่มีค่าใช้จ่าย (open source หรือ free tier) ทั้งหมด เป้าหมายคือให้ผู้ฝึกงานได้ฝึกพัฒนาระบบแบบ full-stack ครบวงจร ตั้งแต่การออกแบบฐานข้อมูล การเขียน API การเชื่อมต่อบริการภายนอก ไปจนถึงการนำระบบขึ้นใช้งานจริง

## 1.2 สิ่งที่ผู้ฝึกงานจะได้เรียนรู้

- การพัฒนาเว็บแบบ full-stack ด้วย Next.js (ฝั่งหน้าเว็บ) และ Express (ฝั่งเซิร์ฟเวอร์)

- การออกแบบฐานข้อมูลเชิงสัมพันธ์ และการใช้ ORM (Prisma) กับ PostgreSQL

- การออกแบบและเขียน REST API พร้อมระบบยืนยันตัวตนด้วย JWT

- การเชื่อมต่อ API ภายนอก เช่น แผนที่ การคำนวณเส้นทาง และการแจ้งเตือน

- การใช้ Git ในการควบคุมเวอร์ชันและการทำงานร่วมกัน

- การเขียนเอกสารทางเทคนิคและการนำเสนอผลงาน

# 2. ขอบเขตของงาน (Scope)

## 2.1 สิ่งที่อยู่ในขอบเขต

- ระบบสมาชิก: สมัครสมาชิก เข้าสู่ระบบ และจัดการโปรไฟล์

- แคตตาล็อกสินค้า: แสดงสินค้า หมวดหมู่ และการค้นหา

- ตะกร้าสินค้าและการสั่งซื้อ (checkout)

- การบันทึกการชำระเงินแบบเก็บเงินปลายทาง (COD) และโอนเงินพร้อมแนบสลิป

- การจัดส่ง: แปลงที่อยู่เป็นพิกัด คำนวณเส้นทางและระยะทาง และแสดงบนแผนที่

- การติดตามสถานะคำสั่งซื้อและการแจ้งเตือน

- แดชบอร์ดสำหรับแอดมิน/ร้านค้าในการจัดการสินค้าและคำสั่งซื้อ

## 2.2 สิ่งที่อยู่นอกขอบเขต

- การเชื่อมต่อช่องทางชำระเงินจริงที่ตัดเงินอัตโนมัติ (payment gateway, การ generate QR ตัดเงิน) — โปรเจกต์นี้บันทึกสถานะการชำระเงินเท่านั้น

- ระบบบัญชี ภาษี หรือใบกำกับภาษีแบบเต็มรูปแบบ

- แอปพลิเคชันมือถือแบบ native (ทำเป็นเว็บ responsive ก่อน)

- ระบบ marketplace หลายผู้ขายขนาดใหญ่ (เริ่มจากร้านเดียวก่อน)

# 3. เทคโนโลยีที่กำหนดให้ใช้

ทุกเครื่องมือและบริการด้านล่างนี้สามารถใช้งานได้โดยไม่มีค่าใช้จ่ายในช่วงพัฒนา (open source หรือ free tier)

| **ส่วนของระบบ** | **เทคโนโลยี** | **หมายเหตุ** |
| --- | --- | --- |
| หน้าเว็บ (Frontend) | Next.js / React | เว็บแบบ responsive |
| เซิร์ฟเวอร์ (Backend) | Node.js + Express | REST API |
| ORM | Prisma | type-safe และทำ migration ง่าย |
| ฐานข้อมูล | PostgreSQL + PostGIS | PostGIS สำหรับการคำนวณเชิงพื้นที่ |
| ยืนยันตัวตน | JWT (jsonwebtoken) | เก็บ token ฝั่ง client |
| ตรวจสอบข้อมูล | zod | validate ข้อมูลที่รับเข้า API |
| เก็บไฟล์รูป | Cloudinary (free tier) | เก็บรูปสินค้าและสลิปโอนเงิน |
| ควบคุมเวอร์ชัน | Git + GitHub/GitLab | บังคับใช้ตลอดโปรเจกต์ |

# 4. สถาปัตยกรรมระบบ

ระบบแบ่งออกเป็น 3 ชั้นหลัก ดังนี้

- **ชั้นนำเสนอ (Presentation): **เว็บฝั่งลูกค้าและแดชบอร์ดแอดมิน พัฒนาด้วย Next.js เรียกใช้งานผ่าน REST API

- **ชั้นประมวลผล (Application): **Express API จัดการ business logic การยืนยันตัวตน และเชื่อมต่อบริการภายนอก

- **ชั้นข้อมูล (Data): **PostgreSQL พร้อมส่วนขยาย PostGIS สำหรับเก็บพิกัดและคำนวณเชิงพื้นที่

บริการภายนอกแบบไม่มีค่าใช้จ่ายที่ระบบเชื่อมต่อ:

| **ฟังก์ชัน** | **บริการที่ใช้** | **ลักษณะการใช้ฟรี** |
| --- | --- | --- |
| แผนที่และแปลงที่อยู่เป็นพิกัด | Longdo Map API / OpenStreetMap | เริ่มต้นใช้งานฟรี (เหมาะกับที่อยู่ไทย) |
| คำนวณเส้นทาง / ระยะทาง | OSRM (ติดตั้งเองด้วย Docker) | ฟรีจริง ไม่จำกัดจำนวนครั้ง |
| แสดงแผนที่บนหน้าเว็บ | Leaflet / MapLibre | open source |
| แจ้งเตือนลูกค้า | LINE Messaging API / อีเมล | ข้อความตอบกลับฟรี / free tier |
| เก็บไฟล์รูปและสลิป | Cloudinary | free tier |

# 5. ความต้องการเชิงฟังก์ชัน (Functional Requirements)

ใช้รหัสอ้างอิงตามกลุ่มผู้ใช้งาน เพื่อให้ติดตามและประเมินผลได้ง่าย

## 5.1 ฝั่งลูกค้า

- **FR-C-01: **สมัครสมาชิก เข้าสู่ระบบ และแก้ไขข้อมูลส่วนตัว

- **FR-C-02: **ดูรายการสินค้า หมวดหมู่ และค้นหาสินค้า

- **FR-C-03: **เพิ่ม/ลบสินค้าในตะกร้า และปรับจำนวน

- **FR-C-04: **กรอกที่อยู่จัดส่ง และปักหมุดตำแหน่งบนแผนที่

- **FR-C-05: **สั่งซื้อสินค้า และเลือกวิธีชำระเงิน (COD หรือ โอนเงิน)

- **FR-C-06: **อัปโหลดสลิปการโอนเงิน (กรณีเลือกโอน)

- **FR-C-07: **ติดตามสถานะคำสั่งซื้อและสถานะการจัดส่ง

- **FR-C-08: **รับการแจ้งเตือนเมื่อสถานะคำสั่งซื้อเปลี่ยนแปลง

## 5.2 ฝั่งแอดมิน / ร้านค้า

- **FR-A-01: **จัดการสินค้า (เพิ่ม แก้ไข ลบ และอัปโหลดรูป)

- **FR-A-02: **จัดการหมวดหมู่สินค้า

- **FR-A-03: **ดูรายการคำสั่งซื้อทั้งหมดและรายละเอียด

- **FR-A-04: **ตรวจสอบสลิปและยืนยันการชำระเงิน (เปลี่ยนสถานะเป็นชำระแล้ว)

- **FR-A-05: **ปรับปรุงสถานะการจัดส่งของแต่ละคำสั่งซื้อ

- **FR-A-06: **ดูเส้นทางและระยะทางการจัดส่งบนแผนที่

- **FR-A-07: **แดชบอร์ดสรุปยอดขายและจำนวนคำสั่งซื้อเบื้องต้น

## 5.3 ระบบจัดส่ง

- **FR-D-01: **แปลงที่อยู่ลูกค้าเป็นพิกัด (geocoding)

- **FR-D-02: **คำนวณเส้นทาง ระยะทาง และเวลาโดยประมาณจากร้านถึงลูกค้า

- **FR-D-03: **แสดงเส้นทางบนแผนที่ในหน้าแอดมิน

- **FR-D-04: **(ส่วนเสริม) จัดลำดับจุดส่งหลายจุดให้ระยะทางสั้นที่สุด

## 5.4 การชำระเงิน

- **FR-P-01: **บันทึกวิธีการชำระเงินและสถานะของแต่ละคำสั่งซื้อ

- **FR-P-02: **กรณี COD: สถานะเป็น 'รอชำระ' จนกว่าจะจัดส่งและเก็บเงินสำเร็จ

- **FR-P-03: **กรณีโอนเงิน: ลูกค้าอัปโหลดสลิป แอดมินตรวจสอบและยืนยันเป็น 'ชำระแล้ว'

**หมายเหตุ: ***โปรเจกต์นี้ไม่เชื่อมต่อระบบตัดเงินจริง การชำระเงินเป็นเพียงการบันทึกและยืนยันสถานะโดยแอดมินเท่านั้น*

# 6. การออกแบบฐานข้อมูล

ให้ผู้ฝึกงานออกแบบ ER diagram และเขียน schema ด้วย Prisma โดยอย่างน้อยควรมีตารางหลักดังต่อไปนี้ (ผู้ฝึกงานสามารถเพิ่มฟิลด์หรือตารางได้ตามความเหมาะสม)

- **User: **ผู้ใช้งาน (role: customer / admin), อีเมล, รหัสผ่านที่เข้ารหัส

- **Address: **ที่อยู่จัดส่งของผู้ใช้ พร้อมพิกัด latitude / longitude

- **Category: **หมวดหมู่สินค้า

- **Product: **สินค้า: ชื่อ ราคา จำนวนคงเหลือ รูปภาพ หมวดหมู่

- **Cart / CartItem: **ตะกร้าสินค้าและรายการในตะกร้า

- **Order / OrderItem: **คำสั่งซื้อและรายการสินค้าในคำสั่งซื้อ

- **Payment: **วิธีชำระเงิน (COD / TRANSFER), สถานะ (PENDING / PAID), ลิงก์สลิป

- **Shipment: **ข้อมูลการจัดส่ง: สถานะ พิกัดปลายทาง ระยะทาง เวลาโดยประมาณ

**งานที่ต้องส่ง: **ER diagram, ไฟล์ schema.prisma และไฟล์ migration ที่ใช้งานได้จริง

# 7. โครงสร้างโปรเจกต์ที่แนะนำ

เนื่องจาก Express ไม่ได้กำหนดโครงสร้างให้ ผู้ฝึกงานควรจัดวางโค้ดฝั่งเซิร์ฟเวอร์แบบแบ่งชั้น (layered) ดังนี้

```
src/
  routes/        นิยาม endpoint ของแต่ละทรัพยากร
  controllers/   รับ request และส่งต่อให้ service
  services/      business logic หลักของระบบ
  middlewares/   auth (JWT), การ validate, การจัดการ error
  prisma/        schema.prisma และ Prisma client
  utils/         ฟังก์ชันช่วยเหลือทั่วไป
  app.js         จุดเริ่มต้นของแอปพลิเคชัน
```

ฝั่ง frontend ใช้โครงสร้างมาตรฐานของ Next.js โดยแยกส่วน components, การเรียก API และการจัดการ state ออกจากกันให้ชัดเจน
